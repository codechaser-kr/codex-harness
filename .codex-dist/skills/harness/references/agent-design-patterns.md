# Agent Design Patterns (Codex용 역할 분리 / 위임 기준)

이 문서는 범용 하네스 시스템을 설계할 때  
역할을 어떻게 나누고, 언제 위임하며, 어떤 구조를 선택해야 하는지 정의한다.

이 문서는 단순 참고가 아니라,  
Codex 환경에서 역할 기반 하네스를 실제로 설계하고 운영하기 위한 기준이다.

또한 이 문서는 `Codex 중심 메타 프레임워크`의 설계 원칙을 담는다.

핵심 전제는 다음과 같다.

- 코드베이스 탐색이 하네스 설계의 1차 입력이다.
- 파일명, 디렉토리명, manifest는 탐색을 시작하는 참고 정보다.
- 역할 분리, 패턴 선택, drift 판단은 탐색 결과 위에서만 의미가 있다.

---

## 0. 메타 프레임워크 개념 모델

Codex 중심 메타 프레임워크에서 핵심 개념은 다음과 같다.

- `탐색 결과`: 코드베이스를 읽어 수집한 근거 묶음
- `역할 팀`: 탐색 결과를 바탕으로 판단을 나누는 실행 단위
- `운영 모드`: 신규 구축 / 기존 확장 / 운영 유지보수
- `아키텍처 패턴`: 파이프라인, 생성-검증, 팬아웃/팬인, 오케스트레이션 중심 구조
- `산출물`: `.codex/skills/*`, `.harness/reports/*`, `.harness/logs/*`
- `drift`: 산출물, 역할 설명, 운영 규칙이 서로 어긋나는 상태

중요한 점은, 이 순서가 거꾸로 되면 안 된다는 것이다.

- 먼저 탐색 결과가 있어야 하고
- 그 다음 역할 팀과 패턴을 고르고
- 마지막에 산출물을 만든다

즉 산출물은 출발점이 아니라 최종 표현이다.

---

## 1. 역할 분리의 목적

역할을 나누는 이유는 단순히 파일 수를 늘리기 위해서가 아니다.

역할 분리의 목적은 다음과 같다.

- 서로 다른 사고방식을 분리한다
- 한 역할이 다루는 컨텍스트를 줄인다
- 판단 단위를 명확하게 만든다
- 필요한 경우 하위 역할에 위임 가능하게 만든다
- 결과를 사람이 더 쉽게 검토하게 만든다

즉 역할 분리는 구조를 복잡하게 만드는 것이 아니라  
👉 **판단을 더 명확하게 만드는 것**이어야 한다.

---

## 2. 역할 분리 기준

역할을 나눌지 여부는 다음 4가지 기준으로 판단한다.

### 1) 전문성 (Specialization)

서로 다른 관점이나 사고방식이 필요한가?

예:
- 저장소 분석
- 구조 설계
- QA 질문 설계
- 검증

이처럼 사고방식이 다르면 역할을 분리할 가치가 높다.

---

### 2) 병렬성 (Parallelism)

서로 독립적으로 다뤄도 되는가?

예:
- 큰 모노레포에서 서로 다른 영역의 기초 분석
- 구조 설계와 일부 QA 초안 작성

독립성이 높으면 분리 후 위임하기 쉽다.

---

### 3) 컨텍스트 크기 (Context Size)

한 역할에 너무 많은 책임이 몰려 있는가?

예:
- 저장소 분석 + 설계 + 생성 + 검증을 한 스킬이 모두 담당

이 경우 결과 품질이 약해지고 유지보수가 어려워진다.

---

### 4) 재사용성 (Reusability)

다른 프로젝트에서도 같은 역할을 반복 사용할 수 있는가?

예:
- domain-analyst
- validator
- qa-designer

이런 역할은 범용 생성기에서 기본 역할로 두기 좋다.

### 추가 전제: 탐색 결과가 먼저 있어야 한다

역할 분리 판단은 추측으로 하면 안 된다.

- 어떤 역할이 필요한가
- 어디까지 분리해야 하는가
- 어떤 패턴이 맞는가

이 질문은 모두 탐색 결과가 나온 뒤에만 판단해야 한다.

---

## 3. 역할을 분리하지 말아야 하는 경우

다음과 같은 경우에는 억지로 분리하지 않는다.

- 입력과 출력이 지나치게 작다
- 실제로는 한 번의 판단으로 끝나는 작업이다
- 분리해도 책임 경계가 불분명하다
- 위임 비용이 이득보다 크다

예:
- 아주 작은 라이브러리에서 orchestrator를 별도로 두는 경우
- 단일 파일 CLI에서 skill-scaffolder와 architect를 굳이 분리하는 경우

핵심 원칙:

👉 **분리는 품질 향상으로 이어질 때만 한다**

---

## 4. 위임 가능한 역할의 조건

Codex의 서브에이전트 활용 관점에서,  
어떤 역할을 독립 판단 단위로 위임하려면 다음 조건이 좋다.

### 1) 입력이 분명하다
무엇을 읽고 판단할지 명확해야 한다.

예:
- domain-analysis.md
- harness-architecture.md
- 저장소 루트 구조

### 2) 출력이 분명하다
무엇을 남겨야 하는지 명확해야 한다.

예:
- qa-strategy.md
- orchestration-plan.md
- 구조 보완 제안

### 3) 역할의 목적이 한 문장으로 설명 가능하다
목적이 흐리면 위임도 흐려진다.

### 4) 사람이 결과를 검토할 수 있다
위임 결과가 파일/리포트로 남지 않으면 관리가 어렵다.

---

## 5. 주요 아키텍처 패턴

범용 하네스에서는 아래 패턴을 기본으로 고려한다.

---

### 1) 파이프라인 (Pipeline)

예:
- domain-analyst → harness-architect → skill-scaffolder → validator

특징:
- 순차 의존성이 강하다
- 범용 하네스 1차 단계의 기본 패턴이다

언제 쓰는가:
- 앞 단계 출력이 뒤 단계의 핵심 입력일 때
- 구조를 안정적으로 만들고 싶을 때
- 탐색 결과를 해석과 산출물로 순차 변환할 때

---

### 2) 생성-검증 (Generate-Validate)

예:
- skill-scaffolder → validator

특징:
- 생성 결과의 최소 품질을 바로 점검할 수 있다
- 하네스 보강 루프에 적합하다

언제 쓰는가:
- 파일 생성 직후 검증이 필요할 때
- 구조적 누락과 약한 설명을 잡고 싶을 때

---

### 3) 팬아웃 / 팬인 (Fan-out / Fan-in)

예:
- 여러 하위 영역 분석 → 하나의 architecture로 통합

특징:
- 대규모 저장소에서 유용하다
- 병렬 판단 단위를 만들 수 있다

언제 쓰는가:
- 모노레포처럼 분석 범위가 클 때
- 서로 다른 하위 영역을 따로 봐야 할 때
- 탐색 결과를 하위 경계별로 병렬 정리한 뒤 다시 통합해야 할 때

주의:
- 범용 하네스 1차 단계에서는 과도한 병렬화보다 단순성을 우선한다

---

### 4) 전문가 선택 (Expert Pool)

예:
- 프로젝트 유형에 따라 다른 역할 구성을 선택

특징:
- 모든 프로젝트에 동일한 풀셋을 강제하지 않는다
- examples와 연결되면 유용하다

언제 쓰는가:
- 프론트엔드 앱 / CLI / 라이브러리 / 모노레포처럼
  역할 구성이 달라져야 할 때

---

### 5) 오케스트레이션 중심 구조

예:
- orchestrator가 전체 흐름과 위임 구조를 조정

특징:
- 역할 간 입력/출력 연결이 핵심이다
- “누가 언제 무엇을 보고 판단하는가”를 명확히 만든다

언제 쓰는가:
- 여러 역할을 함께 운용할 때
- 생성 흐름이 길어질 때

---

## 6. 패턴 선택 기준

패턴은 취향으로 고르지 않는다.  
현재 저장소의 경계 수, 검증 비용, 반복성, 기존 하네스 상태를 기준으로 선택한다.

여기서 말하는 경계 수와 검증 비용은 반드시 탐색 결과에서 나온 근거여야 한다.

### 1) 신규 구축

- 기본값은 `파이프라인 + 생성-검증` 조합이다.
- 이유:
  - 처음에는 입력/출력 연결을 단순하게 잡는 것이 중요하다.
  - 분석 결과가 구조 설계와 스킬 생성 기준으로 바로 이어져야 한다.
  - validator가 초반부터 최소 구조 누락을 잡아야 drift가 줄어든다.

### 2) 기존 확장

- 기본값은 `파이프라인 + 부분 재진입`이다.
- 이유:
  - 이미 있는 구조를 전부 다시 만들기보다, 어디부터 다시 들어갈지 정하는 것이 중요하다.
  - domain-analyst, harness-architect, qa-designer 중 어느 역할을 먼저 재진입시킬지 판단해야 한다.

### 3) 운영 유지보수

- 기본값은 `오케스트레이션 중심 구조 + 생성-검증`이다.
- 이유:
  - 이 단계의 핵심은 새 파일 생성보다 drift 감지와 피드백 루프 유지다.
  - validator와 orchestrator가 구조 회귀를 가장 빨리 드러낼 수 있다.

### 4) 팬아웃 / 팬인을 선택할 때

- 하위 영역 수가 많고 각 영역이 비교적 독립적일 때만 선택한다.
- 팬아웃 전에는 반드시 공통 비교 축을 먼저 정한다.
- 팬아웃 후에는 architecture 또는 orchestration 같은 단일 통합 산출물로 다시 모아야 한다.

### 5) 전문가 선택(Expert Pool)을 선택할 때

- 저장소 유형 차이가 커서 역할 세트 자체가 달라져야 할 때 선택한다.
- 단순히 기술 스택이 다르다는 이유만으로 풀을 나누지 않는다.
- 실행 모델, 검증 비용, 운영 경계가 달라질 때만 풀 분리를 고려한다.

---

## 7. 범용 하네스의 권장 기본 역할

범용 하네스 1차 단계에서는 다음 6개 역할을 기본으로 한다.

- domain-analyst
- harness-architect
- skill-scaffolder
- qa-designer
- orchestrator
- validator

이 구조는 기본값이지 절대 규칙은 아니다.

프로젝트가 단순하면 줄일 수 있고,  
프로젝트가 복잡하면 이후 확장 역할을 추가할 수 있다.

---

## 8. 역할 축소 기준

다음과 같은 경우에는 역할을 줄일 수 있다.

- 저장소 규모가 작다
- 흐름이 단순하다
- QA 질문이 많지 않다
- 오케스트레이션이 사실상 필요 없다

역할 축소는 “파일 수 줄이기”가 아니라  
입력/출력과 판단 기준이 실제로 하나로 합쳐져도 되는지 보고 결정한다.

예:
- 단순 CLI
- 작은 유틸리티 패키지
- 구조가 거의 고정된 라이브러리

이 경우:
- orchestrator 생략 가능
- skill-scaffolder 생략 가능

---

## 9. 역할 확장 기준

다음과 같은 경우에는 역할을 늘리거나 보조 자산을 붙일 수 있다.

- 반복되는 handoff가 로그에 누적된다
- 특정 경계의 검증 비용이 다른 영역보다 유독 크다
- 저장소가 다중 런타임, 다중 배포 단위를 가진다
- 기존 6역할만으로는 중요한 판단 기준이 계속 뭉개진다
- 저장소 규모가 크다
- 분석 축이 많다
- 프로젝트 특화 하네스로 발전할 가능성이 높다
- 이후 expected-state / diff / scenario 레이어가 붙을 가능성이 높다

확장할 때도 원칙은 같다.

- 입력이 분명해야 한다
- 출력이 분명해야 한다
- 기존 역할과 책임 충돌이 없어야 한다
- 사람이 결과를 검토할 수 있어야 한다

예시 확장 역할:
- expected-state-designer
- diff-designer
- scenario-runner

단, 이들은 범용 하네스 1차 단계의 기본 역할은 아니다.

---

## 10. 역할 간 연결 원칙

역할이 여러 개일수록,  
누가 누구의 출력을 입력으로 받는지 명확해야 한다.

권장 연결은 다음과 같다.

- domain-analyst → domain-analysis.md
- harness-architect ← domain-analysis.md
- harness-architect → harness-architecture.md
- skill-scaffolder ← harness-architecture.md
- qa-designer ← domain-analysis.md + harness-architecture.md
- orchestrator ← 주요 리포트 전체
- validator ← 전체 구조와 리포트

즉 역할 분리는 파일 기반 연결과 함께 가야 한다.

---

## 11. 사람이 검토 가능한 구조를 우선하라

역할을 잘 나눴더라도,  
결과가 사람이 읽고 수정할 수 없으면 범용 하네스로서 가치가 약해진다.

따라서 항상 다음을 우선한다.

- 리포트로 남길 것
- 입력/출력을 파일 단위로 명확히 할 것
- 추정은 추정이라고 적을 것
- 자동화보다 구조적 명확성을 우선할 것

---

## 12. 핵심 요약

- 역할 분리는 판단 명확화를 위한 것이다
- 위임은 입력/출력이 분명할 때 가치가 크다
- 범용 하네스의 기본 패턴은 파이프라인 + 생성-검증이다
- 병렬화는 큰 저장소에서만 신중하게 사용한다
- 역할 수는 많을수록 좋은 것이 아니다
- 결과는 항상 사람이 검토 가능한 구조로 남겨야 한다
