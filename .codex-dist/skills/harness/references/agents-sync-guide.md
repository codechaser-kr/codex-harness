# AGENTS Sync Guide

이 문서는 기존 저장소에 `AGENTS.md`가 있을 때, 실행 하네스가 이를 어떻게 다뤄야 하는지 정의한다.

`Codex` 기준에서 `AGENTS.md`는 가장 가까운 상위 컨텍스트 문서다.  
하지만 현재 하네스의 본체는 `.codex/skills/*`와 `.harness/*`이며, `AGENTS.md`는 이를 덮어쓰는 문서가 아니라 상위 운영 계약과 진입 규칙을 정렬하는 문서로 다룬다.

---

## 핵심 전제

- 기존 `AGENTS.md`는 자동으로 덮어쓰지 않는다.
- 먼저 읽고, 현재 하네스 구조와 정렬되는지 감사한다.
- 충돌이 작으면 보강하고, 충돌이 크면 명시적 재구성을 제안한다.
- `AGENTS.md`는 상위 운영 계약을 담고, 세부 실행 구조는 `.codex/skills/*`와 `.harness/*`가 담당한다.

---

## 1. 언제 확인하는가

- `harness-init.sh`를 돌리기 전
- `harness-update.sh`로 기존 확장을 시작하기 전
- `harness-verify.sh`에서 drift를 볼 때

즉 `AGENTS.md`는 생성 후 참고 문서가 아니라, 하네스 현황 감사의 일부로 다뤄야 한다.

---

## 2. 무엇을 감사하는가

기존 `AGENTS.md`가 있으면 최소한 다음을 본다.

- 상위 운영 원칙이 있는가
- 역할 진입점이나 우선순위 규칙이 있는가
- 금지/필수 작업 규칙이 있는가
- 현재 하네스와 충돌하는 역할 체계가 있는가
- `init/update/verify` 또는 이에 대응되는 운영 흐름이 이미 있는가

핵심은 파일 존재 여부가 아니라, 현재 하네스와 운영 계약이 얼마나 정렬되는지다.

---

## 3. 판단 결과

감사 결과는 다음 넷 중 하나로 정리한다.

### 정렬됨

- 현재 `AGENTS.md`가 하네스 운영 방향과 크게 충돌하지 않는다.
- 필요한 경우 시작 진입 역할, `init/update/verify` 진입 규칙만 보강한다.

### 보강 필요

- 큰 충돌은 없지만 상위 운영 계약이 약하다.
- 현재 하네스 구조를 기준으로 `AGENTS.md`에 상위 진입점, 운영 모드, 재구성 원칙을 보강한다.

### 충돌

- 기존 `AGENTS.md`가 다른 역할 체계나 다른 오케스트레이션 규칙을 강하게 전제한다.
- 자동 병합하지 않고, 어떤 규칙이 충돌하는지 먼저 설명한 뒤 사용자 판단을 구한다.

### 재구성 필요

- `AGENTS.md`와 하네스 구조가 모두 부분적으로 깨졌거나 서로 다른 운영 모델을 말한다.
- update로 봉합하지 말고, 기존 구조 정리와 명시적 재구성을 우선 제안한다.

---

## 4. 보강 원칙

기존 `AGENTS.md`를 보강할 때는 다음만 상위 문서에 남긴다.

- 하네스 진입점
- 운영 모드
- drift / 재구성 기준
- 상위 우선순위 규칙

다음은 `AGENTS.md`가 아니라 하위 자산에 남긴다.

- 역할별 상세 지침
- QA 질문 세트
- 보고서 세부 구조
- 로그 운영 세부 규칙

즉 `AGENTS.md`는 상위 인덱스와 운영 계약만 다루고, 세부 내용은 기존 하네스 자산으로 내려 보낸다.

---

## 5. 다른 레퍼런스와의 연결

- `reference-writing-guide.md`: 내부 설계 레퍼런스의 공통 용어와 구조 기준을 제공한다.
- `agent-design-patterns.md`: 실행 단위와 패턴 선택 기준이 `AGENTS.md`의 상위 운영 계약에 어떻게 반영되는지 연결한다.
- `orchestrator-template.md`: `AGENTS.md`에서 선언한 진입 규칙이 실제 팀 흐름으로 어떻게 번역되는지 연결한다.
