# Exploration Model

`harness`는 코드베이스 탐색에서 직접 수집한 근거를 공통 입력으로 사용한다.
이 문서는 탐색 결과를 어떤 의미 단위로 정리할지 정의한다.

## 목적

- 언어나 프레임워크 라벨보다 실제 파일·경로·경계를 먼저 수집한다.
- 역할 팀과 `.harness/reports/*` 문서가 같은 근거를 공유하게 만든다.
- 해석 문단이 코드베이스 근거보다 앞서지 않게 한다.

## 최소 탐색 단위

탐색은 최소한 다음 다섯 가지를 수집한다.

1. 대표 진입점
2. 주요 코드 경계
3. 테스트 및 검증 자산
4. 설정 및 실행 경로
5. 저장소 고유 용어 단서

## 탐색 상태

탐색 결과는 수집 밀도와 연결성에 따라 다음 세 상태로 해석한다.

- `초기`: 대표 진입점, 코드 경계, 테스트/검증 자산 중 대부분이 비어 있거나 fallback 문장만 있는 상태
- `제한적`: 일부 경로와 근거는 수집됐지만 경계 연결이나 검증 자산 설명이 아직 약한 상태
- `충분`: 대표 진입점, 주요 코드 경계, 테스트/검증 자산, 설정/실행 경로, 저장소 고유 용어가 실제 경로 근거와 함께 연결된 상태

이 상태는 프로젝트 크기 분류가 아니라 현재 탐색이 얼마나 실제 판단 입력으로 쓸 수 있는지 보여주는 공통 지표다.

## 정리 방식

- 탐색 결과는 파일명 추정보다 실제 파일·경로 목록으로 먼저 정리한다.
- 경계는 가능하면 저장소의 실제 코드 단위로 묶는다.
- 문서 용어는 README, 운영 문서, 도메인 설명 문서 같은 원문 경로와 함께 남긴다.
- 해석과 운영 규칙은 탐색 결과 뒤에 온다.
- `초기` 상태에서는 질문과 추가 탐색 경로를 함께 남기고, `제한적` 상태에서는 어떤 근거가 더 필요한지 적고, `충분` 상태에서는 운영 규칙과 verify 기준을 더 구체적으로 연결한다.

## 사용 위치

- `harness-explore.sh`는 이 모델을 따라 탐색 근거 문서를 만든다.
- `harness-init.sh`와 `harness-update.sh`는 이 결과를 공통 입력으로 읽는다.
- `harness-verify.sh`는 탐색 상태에 따라 질문 우선, 선택 갱신, 구체 검증 강도를 나눠 적용한다.
- `harness-verify.sh`는 탐색 근거와 문서 해석의 연결을 점검한다.
