# Exploration Model

`harness`는 자동 탐색 결과를 최종 판단 엔진으로 사용하지 않는다.
이 문서는 `exploration-notes.md`를 약한 탐색 메모로 유지하고, 후속 역할이 실제 저장소와 사용자 입력을 다시 읽어 최종 판단을 내리게 하는 기준을 정의한다.

## 목적

- 입력 메모를 최종 판단 근거에서 내린다.
- 역할 팀이 실제 저장소와 사용자 입력을 다시 읽게 만든다.
- 해석 문단이 약한 메모를 복사하는 방식으로 작성되지 않게 한다.
- 입력 메모가 최종 판단이 아니라, 입력 부족 신호와 추가 확인 질문을 남기는 문서가 되게 만든다.

## 입력 메모의 책임

`exploration-notes.md`는 최소한 다음 네 가지를 남긴다.

1. 자동 수집을 최종 판단 근거로 쓰지 않는다는 전제
2. 현재 사용자 입력이 있는지 없는지
3. 역할 팀이 문서를 다시 써야 한다는 메모
4. 다음에 확인할 질문

## 입력 상태

입력 상태는 자동 수집량이 아니라 사용자 입력 준비 상태 기준으로 해석한다.

- `초기`: `project-setup.md`나 사용자 답변이 없어, 입력 메모만 있는 상태
- `제한적`: `project-setup.md` 또는 사용자 답변은 있어 방향을 좁힐 수 있지만, 여전히 역할 스킬의 저장소 재해석이 필요한 상태

`exploration-notes`만으로 `충분` 상태가 되지 않는다. 충분한 상태는 역할 스킬이 최종 문서를 직접 쓴 뒤에야 판단할 수 있다.

## 정리 방식

- 자동 탐색은 파일 경로를 많이 모으는 대신, “지금 자동 판단을 확정하지 않는다”는 사실을 먼저 남긴다.
- 저장소의 실제 시작점, 경계, 테스트 경로, 운영 경로는 저장소 입력 문서 담당 역할과 후속 역할이 다시 읽어 최종 문서에 적는다.
- `초기` 상태에서는 사용자 질문과 `project-setup.md` 작성을 앞에 둔다.
- `제한적` 상태에서는 사용자 입력을 바탕으로 역할 스킬이 저장소를 직접 다시 읽게 한다.

## 사용 위치

- `harness-explore.sh`는 이 모델을 따라 약한 탐색 메모를 만든다.
- `harness-init.sh`와 `harness-update.sh`는 이 메모와 `project-setup.md` 상태를 읽는다.
- 시작 진입 역할은 이 문서만으로 시작 역할을 단정하지 않는다.
- `harness-verify.sh`는 입력 메모의 풍부함보다, 역할 재작성과 운영 계약 정렬 여부를 더 앞에서 본다.
