# Exploration Model

`harness`는 코드베이스 탐색에서 직접 수집한 근거를 공통 입력으로 사용한다.
이 문서는 자동 탐색 결과를 어떤 후보 단서로 정리하고, 후속 역할이 그것을 어떻게 다시 읽어야 하는지 정의한다.

## 목적

- 언어나 프레임워크 라벨보다 실제 저장소 근거를 먼저 수집한다.
- 역할 팀과 `.harness/reports/*` 문서가 같은 근거를 공유하게 만든다.
- 해석 문단이 코드베이스 근거보다 앞서지 않게 한다.
- 탐색 문서가 최종 판단이 아니라 후속 해석을 위한 공통 입력이 되게 만든다.

## 최소 탐색 단위

탐색은 최소한 다음 다섯 가지를 수집한다.

1. 대표 시작점 후보
2. 관련 코드 경로 후보
3. 테스트 및 검증 자산
4. 설정 및 실행 경로
5. 저장소 용어 단서

## 탐색 상태

탐색 결과는 수집 밀도와 연결성에 따라 다음 세 상태로 해석한다.

- `초기`: 대표 시작점 후보, 관련 코드 경로, 테스트/검증 자산 중 대부분이 비어 있거나 fallback 문장만 있는 상태
- `제한적`: 일부 경로와 근거는 수집됐지만 실제 흐름과 책임을 바로 해석하기엔 약한 상태
- `충분`: 대표 시작점 후보, 관련 코드 경로, 테스트/검증 자산, 설정/실행 경로, 저장소 용어 단서가 실제 경로 근거와 함께 연결된 상태

이 상태는 프로젝트 크기 분류가 아니라 현재 탐색이 얼마나 실제 판단 입력으로 쓸 수 있는지 보여주는 공통 지표다.

## 정리 방식

- 탐색 결과는 파일명 추정보다 실제 시작점과 함께 읽힌 코드 경로를 먼저 정리한다.
- 관련 코드 경로는 저장소의 실제 코드 단위와 소비 관계를 해석하기 위한 후보로 남긴다.
- 대표 진입점 후보는 단순히 먼저 발견된 `index.*`를 택하지 않고, 실제 시작 흐름에 가까운 `main.*`, `app.*`, 루트 `src/index.*`, Electron 메인 진입처럼 상위 시작점에 더 큰 가중치를 둔다.
- 코드 경계 후보는 저장소 전체 설정 파일보다 실제 코드가 모이는 디렉토리 경계를 우선 남긴다.
- 대표 진입점은 자동으로 포착한 시작점 후보이며, 실제 사용자 흐름의 시작점과 동일하다고 가정하지 않는다.
- 설정 및 실행 경로는 실행/빌드/배포 후보 단서이며, 실제 운영 경로 여부는 후속 역할이 다시 확인한다.
- 설정 및 실행 경로는 런타임·빌드·배포와 직접 연결된 파일을 우선 남기고, CI 메타 디렉토리나 잠금 파일, 패키지 매니저 dotfile은 기본 후보에서 제외한다.
- 저장소 용어 단서는 문서에서 포착한 용어 후보이며, 실제 도메인 용어는 코드와 함께 다시 검토한다.
- 문서 용어는 README, 패키지 문서, 도메인 설명 문서 같은 원문 경로와 함께 남긴다.
- 해석과 운영 규칙은 탐색 결과 뒤에 온다.
- `초기` 상태에서는 질문과 추가 탐색 경로를 함께 남기고, `제한적` 상태에서는 어떤 흐름과 책임이 중심인지 더 필요한 근거를 적고, `충분` 상태에서는 운영 규칙과 verify 기준을 더 구체적으로 연결한다.

## 사용 위치

- `harness-explore.sh`는 이 모델을 따라 탐색 근거 문서를 만든다.
- `harness-init.sh`와 `harness-update.sh`는 이 결과를 공통 입력으로 읽는다.
- `harness-verify.sh`는 탐색 상태에 따라 질문 우선, 선택 갱신, 구체 검증 강도를 나눠 적용한다.
- `harness-verify.sh`는 탐색 근거와 문서 해석의 연결을 점검한다.
