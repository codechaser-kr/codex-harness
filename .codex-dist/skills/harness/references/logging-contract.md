# Logging Contract

이 문서는 MD-only 하네스에서 세션 기록을 어떤 문서로 남기고 어떻게 유지하는지 정의한다.

핵심 원칙은 로그도 실행 결과를 설명하는 문서여야 한다는 점이다. TSV나 임시 상태 파일보다 사람이 읽고 다음 역할이 바로 이어받을 수 있는 Markdown 기록을 우선한다.

## 핵심 전제

- 세션 기록의 진실원천은 Markdown 문서다.
- 기록의 목적은 통계 수집보다 다음 역할과 다음 세션이 바로 이어받을 수 있게 하는 것이다.
- 로그는 선택 사항이 아니라 실행 완료 조건의 일부다.
- 세션 기록은 append-only 저널과 최신 요약으로 분리한다.

## 필수 로그 문서

- `.harness/logs/session-log.md`
- `.harness/logs/latest-session-summary.md`

상황에 따라 유지할 수 있는 보조 문서:

- `.harness/logs/role-frequency.md`
- `.harness/docs/template-candidates.md`

## session-log.md 계약

`session-log.md`는 세션 단위 저널이다. 같은 세션 안에서는 여러 항목이 누적될 수 있다.

각 세션 기록은 최소한 다음을 포함한다.

- 시각
- 세션 ID
- 상태
- 시작 요청
- 진입점
- 계획 역할
- 호출 역할
- 실행 결과 상태
- 입력 요약
- 출력 요약
- 변경 파일
- 다음 권장 역할
- 남은 위험
- 메모

## latest-session-summary.md 계약

`latest-session-summary.md`는 가장 최근 종료 세션의 현재 상태를 한 번에 읽을 수 있게 정리한 문서다.

최소 포함 항목:

- 세션 ID
- 시작 시각
- 종료 시각
- 기록 수
- 계획 역할
- 호출 역할
- 역할 결과 상태
- 최근 입력 요약
- 최근 출력 요약
- 최근 변경 파일
- 마지막 권장 역할
- 다음 재진입 phase
- 남은 위험
- 다음 시작 전 우선 확인 입력 파일
- 최근 출력 파일
- 세션 종료 메모

## 결과 상태 규약

역할 결과 상태는 아래 중 하나를 사용한다.

- `planned`
- `in_progress`
- `completed`
- `timed_out`
- `failed`

시작 이벤트처럼 아직 실행 결과가 없는 경우에는 상태를 비워 둘 수 있지만, 실제 역할 수행이 기록되는 항목에는 가능한 한 결과 상태를 명시한다.

## 기록 규칙

- 역할 실행이나 문서 변경이 있으면 같은 세션 기록에도 반영한다.
- 비동기 역할은 최종 응답 전에 `completed` 또는 `timed_out`으로 정리한다.
- `timed_out`과 `failed`는 숨기지 말고 남은 위험과 후속 조치로 연결한다.
- 통계 자동화보다 재진입 가독성을 우선한다.

## 완료 조건

하네스 작업을 끝났다고 말하려면 최소한 다음을 만족해야 한다.

- `session-log.md`에 현재 세션의 주요 진행과 종료가 남아 있다.
- `latest-session-summary.md`가 마지막 종료 세션 기준으로 갱신돼 있다.
- 다음 역할 또는 다음 재진입 Phase가 명시돼 있다.
- 다음 시작 전에 다시 읽을 입력과 최근 출력이 남아 있다.

## 다른 레퍼런스와의 연결

- `reentry-rules.md`: 요약 문서에 어떤 재진입 정보를 남겨야 하는지 정의한다.
- `verification-checklist.md`: 로그 문서가 최소 기준을 충족하는지 검토할 때 사용한다.
- `target-evaluation-playbook.md`: 로그를 읽고 운영 가능/재작성 필요/재구성 필요를 판정할 때 사용한다.
