# Meta-System Maturity Guide

이 문서는 `harness`가 단순 생성기를 넘어, 상위 수준의 Codex 중심 하네스 메타시스템으로 성숙했는지 판단하는 기준을 정의한다.

핵심은 파일 수나 문서 존재 여부가 아니라 아래 질문이다.

- 생성된 하네스가 실제로 팀처럼 동작하는가
- 시작점, handoff, 재진입, 종료 기준이 프로젝트마다 다시 살아나는가
- 입력 문서와 메타시스템 문서가 서로 다른 책임을 끝까지 유지하는가
- 프로젝트 내부에서 반복 운영해도 generic scaffold로 되돌아가지 않는가

---

## 1. 성숙도 판단 원칙

- 성숙도 평가는 “구조 생성 성공”이 아니라 “운영 가능한가”를 본다.
- 평가 대상은 전역 스킬, 생성 스크립트, reference 체계, 생성된 로컬 에이전트 설정, 생성된 로컬 스킬, 생성된 보고서 전체다.
- 상위 수준의 메타시스템 성숙도란 문서 톤의 유사성이 아니라, 메타시스템의 중심축이 `에이전트 팀 / 실행 패턴 / 운영 계약`으로 고정돼 있다는 뜻이다.
- 특정 저장소에서 한 번 그럴듯한 보고서를 만든 사례보다, 다른 저장소로 옮겨도 같은 운영 계약을 재현할 수 있는지가 더 중요하다.

---

## 2. 필수 성숙도 축

아래 여섯 축은 모두 충족돼야 “대등 수준에 근접했다”고 본다.

### 1) 실행 팀 우선 구조

- 시작 진입 역할이 실제 시작 엔트리포인트처럼 동작한다.
- 역할 스킬은 초안 보강자가 아니라 각 문서의 최종 작성 책임자로 읽힌다.
- `AGENTS.md`, `.codex/config.toml`, `.codex/agents/*.toml`, `.codex/skills/*`가 같은 팀 계약을 말한다.

### 2) 입력 문서 품질

- `exploration-notes.md`는 자동 판단 보류를 위한 약한 메모로서, 후속 역할이 저장소를 다시 읽어야 한다는 사실을 분명히 남긴다.
- `domain-analysis.md`, `qa-strategy.md`는 저장소 입력 문서로 직접 읽히고, 가이드 문체나 메타 설명이 주제가 되지 않는다.
- 입력 문서는 경로 나열보다 실제 저장소 경계, 흐름, 실패 비용을 우선 설명한다.

### 3) 메타시스템 문서 품질

- `harness-architecture.md`, `orchestration-plan.md`, `team-structure.md`, `team-playbook.md`는 저장소 문서가 아니라 하네스 운영 문서로 읽힌다.
- 각 문서는 역할 소개보다 패턴 선택, 개입 기준, handoff, 재진입, 종료 조건이 먼저 보인다.
- 메타시스템 문서가 입력 문서 내용을 대신 요약하지 않는다.

### 4) 운영 계약 명확성

- 상태 모드, 실행 모드, 실행 패턴, drift / sync / evolve가 선택지가 아니라 실제 계약으로 유지된다.
- 운영 감사 역할은 파일 존재 검사자가 아니라 운영 계약 감사자로 동작한다.
- verify는 구조 검증, 운영 감사 역할은 운영 계약 검증이라는 경계가 분명하다.

### 5) 재사용 가능성

- 특정 프레임워크나 특정 저장소 구조에 과적합되지 않는다.
- 탐색은 휴리스틱 후보 수집에 머무르되, 최종 판단은 역할 스킬이 저장소를 다시 읽도록 설계돼 있다.
- 프로젝트마다 다를 수 있는 expected state, scenario, diff 기준은 로컬 하네스 확장 대상으로 남긴다.

### 6) 품질 비교 가능성

- `without-skill` 대비 시작 역할 판단, 질문 절제, handoff, 저장소 근거 연결, 검증 가능성이 실제로 좋아졌는지 설명할 수 있다.
- “좋아 보인다”가 아니라 “무엇이 어떻게 달라졌는가”를 비교 문장으로 남길 수 있다.

---

## 3. 미성숙 신호

아래 신호가 반복되면 아직 대등 수준이 아니다.

- 생성된 보고서가 분석 결과가 아니라 “어떻게 해야 한다”는 가이드 문체에 머문다.
- `exploration-notes.md`가 다시 경로 후보 나열 문서로 되돌아가고, 역할 스킬의 저장소 재독해 필요성을 약하게 만든다.
- 역할 스킬이 최종 작성자처럼 읽히지 않고, 초안 보정자나 메모 보강자처럼 읽힌다.
- 메타시스템 문서가 역할 카드 소개에 머물고, 실제 시작점과 종료 조건이 약하다.
- 운영 감사 피드백이 재작성 책임 문서를 분명히 지정하지 못한다.
- 특정 저장소에서는 우연히 맞지만, 저장소가 바뀌면 같은 구조가 generic scaffold로 무너진다.

---

## 4. 대등 수준 판정 질문

평가할 때는 아래 질문을 순서대로 본다.

1. 생성 직후 상태에서 “아직 초안 상태”와 “이제 역할 작성이 필요함”이 명확히 드러나는가
2. 역할 작성 이후 보고서가 scaffold가 아니라 최종 문서처럼 읽히는가
3. 입력 문서와 메타시스템 문서가 서로 다른 목적을 끝까지 유지하는가
4. 시작 진입 역할이 다음 시작 역할과 재진입 지점을 바로 제시할 수 있는가
5. 운영 감사 역할이 어떤 문서를 누가 다시 써야 하는지 실행 가능한 수준으로 지정하는가
6. 저장소가 바뀌어도 같은 계약으로 에이전트 팀을 다시 세울 수 있는가

여섯 질문 중 하나라도 반복적으로 실패하면 아직 “대등 수준”으로 보지 않는다.

---

## 5. 현재 생성기에서 우선 봐야 할 개선 포인트

- 입력 문서 품질 저하가 후속 문서 품질 저하로 번지는지
- 역할 스킬 본문이 최종 작성 책임을 충분히 강하게 말하는지
- 메타시스템 문서가 역할 설명보다 운영 계약 문서로 먼저 읽히는지
- README와 reference 체계가 같은 성숙도 기준을 공유하는지
- 스모크 테스트가 구조 회귀만이 아니라 최소한의 입력 품질 회귀를 잡는지

---

## 6. 대등 수준 확인 질문

`대등 수준`에 근접했는지 보려면 아래를 함께 통과해야 한다.

- 타겟 프로젝트마다 새로운 역할 팀이 실제로 설계되는가
- 생성된 역할 스킬이 시작 경로, 요청 분기, 출력 형식, 재진입/종료 규칙까지 담는가
- `team-playbook`, `latest-session-summary`, `logging-policy`가 세션 종료 뒤 다음 재진입을 바로 제시하는가
- 평가 결과가 `입력 문서 / 역할 설계 / 생성 스킬 / 운영 계약` 중 어느 축을 고쳐야 하는지 바로 연결되는가
- 타겟 프로젝트 평가가 한 번의 인상 비평이 아니라, 다시 어떤 phase로 들어갈지 실행 가능한 backlog를 남기는가

---

## 7. 사용 위치

- 시작 진입 역할이 현재 상태를 “운영 가능 / 재작성 필요 / 재구성 필요”로 판단할 때 참고한다.
- 운영 감사 역할이 단순 drift가 아니라 성숙도 부족 신호를 지적할 때 참고한다.
- README에서 현재 범위와 한계를 설명할 때 이 문서 기준으로 말한다.
- 품질 비교나 회고를 남길 때 `quality-evaluation-guide.md`와 함께 사용한다.
- phase 0~7 운영 계약에서는 `Phase 7 품질 비교와 성숙도 평가`의 기준 문서로 사용한다.
