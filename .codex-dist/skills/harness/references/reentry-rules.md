# Reentry Rules

이 문서는 하네스 작업이 한 번에 끝나지 않을 때 어떤 Phase로 되돌아가야 하는지와, 다음 시작을 위해 어떤 정보를 남겨야 하는지 정의한다.

재진입은 예외 처리가 아니라 운영 루프의 일부다.

## 핵심 전제

- 재진입 판단은 문서 품질, 역할 규칙, 운영 기준 중 어디가 약한지에 따라 달라진다.
- 모든 재진입은 `latest-session-summary.md`에 다음 시작 정보로 남아야 한다.
- 재진입은 전체 재생성보다 좁은 Phase 수정부터 검토한다.

## 기본 재진입 축

- 입력이 약하면 `Phase 1`
- 역할 경계가 약하면 `Phase 2`
- agent/skill 생성 결과가 약하면 `Phase 3`
- QA 기준이 약하면 `Phase 4`
- 오케스트레이션과 운영 문서가 약하면 `Phase 5`
- 운영 감사에서 구조 불일치가 남으면 `Phase 6`
- 운영 가능성 판단이 약하면 `Phase 7`

## Phase별 재진입 조건

### Phase 1로 재진입

- `project-setup.md`가 비어 있거나 방향을 좁히지 못한다
- `domain-analysis.md`가 저장소 근거보다 일반론에 가깝다
- 핵심 사용자 문제나 첫 성공 시나리오가 불분명하다

### Phase 2로 재진입

- 역할 이름이 범용적이거나 현재 저장소 용어를 반영하지 못한다
- 다음 역할 흐름이 약하다
- 시작 진입 역할과 중심 조율 역할이 섞여 있다

### Phase 3으로 재진입

- `team-spec.md`와 `.codex/agents/*.toml`, `.codex/skills/*`가 일치하지 않는다
- description이 트리거되기 어렵다
- 역할 스킬의 입력/출력/종료 기준이 빈약하다

### Phase 4로 재진입

- `qa-strategy.md`가 자동/수동 검증을 구분하지 못한다
- QA 역할과 운영 감사 역할의 경계가 약하다
- 실패 비용이 큰 흐름이 품질 전략에 반영되지 않는다

### Phase 5로 재진입

- `orchestration-plan.md`, `team-playbook.md`, `team-structure.md`가 다음 역할 흐름을 분명히 보여주지 못한다
- 입력 문서와 메타시스템 문서의 목적이 섞인다
- 최신 세션 요약이 다음 시작 정보를 충분히 남기지 못한다

### Phase 6으로 재진입

- 운영 감사가 누락/불일치/골격 잔존을 남긴다
- 역할 구조가 확장 가능한 팀으로 읽히지 않는다
- 문서 계층과 생성 결과가 같은 운영 기준을 말하지 않는다

### Phase 7로 재진입

- 구조는 맞지만 운영 가능성을 설명하지 못한다
- `with-skill`과 `without-skill` 비교 관찰이 약하다
- 다음 재진입 판단이 품질 문제와 연결되지 않는다

## latest-session-summary.md에 남겨야 할 재진입 정보

최신 요약에는 최소한 아래가 있어야 한다.

- 다음 시작 역할
- 다음 재진입 phase
- 다음 시작 전 우선 확인 입력 파일
- 최근 출력 파일
- 최근 변경 파일
- 남은 위험 또는 미해결 항목

## 판단 원칙

- 전체를 다시 만드는 것보다 가장 이른 실패 지점으로 되돌아간다.
- 스크립트 성공 여부보다 문서 품질과 역할 계약 충족 여부를 우선한다.
- 재진입 이유는 한 문장으로 설명 가능해야 한다.

## 다른 레퍼런스와의 연결

- `phase-selection-matrix.md`: Phase 선택 기준을 더 자세히 볼 때 사용한다.
- `logging-contract.md`: 최신 세션 요약에 무엇을 남겨야 하는지 정의한다.
- `verification-checklist.md`: 재진입이 필요한 불일치 유형을 검토할 때 사용한다.
