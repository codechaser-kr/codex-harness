# Internal Reference Writing Guide

이 문서는 `references/` 아래 내부 설계 레퍼런스를 어떤 형식과 용어로 작성할지 정의한다.

대상 문서:

- `agent-design-patterns.md`
- `orchestrator-template.md`
- `qa-agent-guide.md`
- `team-examples.md`
- `skill-writing-guide.md`
- `skill-testing-guide.md`
- 이후 추가되는 내부 설계 레퍼런스

이 문서의 목적은 두 가지다.

- 내부 설계 레퍼런스가 서로 다른 말투와 구조로 흩어지지 않게 한다.
- 외부 노출 문서와 달리, 내부 레퍼런스에서는 설계 언어를 더 직접적이고 명확하게 유지한다.

---

## 1. 기본 원칙

- 내부 설계 레퍼런스는 자연스러운 소개문보다 설계 판단과 운영 기준을 우선한다.
- 같은 개념은 같은 용어로 반복한다.
- 설명은 반드시 `왜 이런 기준이 필요한가`를 포함한다.
- 탐색 결과가 모든 구조 판단의 선행 입력이라는 전제를 유지한다.
- 외부 노출 문서에서 완곡하게 바꾼 표현이라도, 내부 레퍼런스에서는 필요하면 설계 용어를 직접 사용한다.

---

## 2. 공통 권장 구조

모든 내부 레퍼런스가 완전히 같은 목차를 가질 필요는 없지만, 아래 흐름을 기본 골격으로 삼는다.

1. 문서 목적
2. 핵심 전제 또는 입력
3. 판단 기준
4. 권장 구조 또는 권장 흐름
5. 피해야 할 실수
6. 다른 레퍼런스/산출물과의 연결

최소한 다음 둘 중 하나는 반드시 문서 안에 드러나야 한다.

- 어떤 결정을 내려야 하는가
- 그 결정을 무엇을 근거로 내리는가

---

## 3. 공통 용어

내부 레퍼런스에서는 다음 용어를 기본 설계 언어로 유지한다.

- `탐색 결과`
- `운영 모드`
- `신규 구축`
- `기존 확장`
- `운영 유지보수`
- `재구성`
- `Agent Teams`
- `Subagents`
- `아키텍처 패턴`
- `drift`
- `handoff`
- `재진입`
- `운영 감사 피드백 루프`

같은 개념을 문서마다 다른 말로 바꾸지 않는다.

예:

- `Agent Teams`를 어떤 문서에서는 `장기 운영 팀`, 다른 문서에서는 `핵심 팀 구조`로 바꾸지 않는다.
- `패턴 선택 기준`을 어떤 문서에서는 `패턴 운영`, 다른 문서에서는 `운영 구조 결정`으로 섞지 않는다.

---

## 4. 서술 방식

- 먼저 결론을 말하고, 그 다음 이유를 적는다.
- 가능하면 명사형 개념 설명보다 판단 문장으로 쓴다.
- 예시는 실제 프로젝트 특화 역할 이름이나 기능 역할 이름을 써서 연결한다.
- 추상어만 반복하지 말고, 저장소 입력 문서 담당 역할, 흐름 조율 역할, 운영 감사 역할 같은 기능 역할과 연결한다.
- `좋다`, `권장한다`만 쓰지 말고, 왜 그런지 drift, 검증 비용, handoff, 재진입 관점으로 설명한다.

좋은 예:

- `기존 확장에서는 전부 재생성하지 말고, drift가 난 역할부터 재진입시킨다. 이유는 기존 구조를 유지한 채 변경 비용을 국소화할 수 있기 때문이다.`

나쁜 예:

- `기존 확장에서는 유연하게 접근한다.`

---

## 5. 외부 문서와의 톤 차이

외부 노출 문서:

- 자연스러운 운영 언어를 우선한다.
- 특정 설계 용어를 과도하게 전면화하지 않는다.

내부 설계 레퍼런스:

- 설계 용어를 숨기지 않는다.
- `Agent Teams / Subagents`, `패턴 선택 기준`, `운영 모드`, `drift` 같은 표현을 직접 쓴다.
- 사람에게 설명하기보다 프레임워크를 설계하고 유지하는 기준으로 쓴다.

## 6. 문서별 기대 초점

### agent-design-patterns

- 역할 분리 기준
- 실행 단위 선택 기준
- 아키텍처 패턴 선택 기준
- 에이전트 정의 구조

### orchestrator-template

- 흐름 조율 역할의 책임
- 운영 모드별 조율 기준
- 패턴 선택 기준
- handoff / 재진입 / 피드백 루프

### qa-agent-guide

- QA가 검토해야 하는 경계
- QA 질문이 구조 판단에 미치는 영향
- 운영 감사 역할 및 흐름 조율 역할과의 관계

### team-examples

- 실제 저장소 타입별 팀 구조 예시
- 어떤 조건에서 역할 수를 줄이거나 늘리는지

### skill-writing-guide

- 스킬 description과 본문이 어떤 판단 단위를 드러내야 하는지
- 입력/출력/협업 기준을 어떻게 쓰는지

### skill-testing-guide

- 무엇을 테스트해야 하는지
- 어떤 회귀를 반드시 막아야 하는지

---

## 7. 회귀 방지 원칙

내부 설계 레퍼런스를 수정할 때는 아래가 깨지지 않는지 확인한다.

- 공통 용어가 문서마다 달라지지 않았는가
- 판단 기준 섹션이 설명보다 약해지지 않았는가
- 실제 역할 이름 또는 기능 역할과의 연결이 사라지지 않았는가
- 외부 문서용 완곡 표현이 내부 레퍼런스까지 퍼지지 않았는가

이 원칙은 스모크 테스트나 정적 확인으로 함께 고정하는 것을 우선한다.
