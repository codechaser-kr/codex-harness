# Skill Testing Guide (Codex용 스킬 검증 기준)

이 문서는 생성된 스킬이 실제로 올바르게 작동하는지 검증하는 방법을 정의한다.

스킬은 작성하는 것으로 끝나지 않는다.  
반드시 테스트를 통해 유효성을 확인해야 한다.

범용 하네스 생성기에서는 특히 다음을 검증해야 한다.

- 스킬이 실제로 트리거되는가
- 스킬이 없는 경우보다 더 나은 결과를 만드는가
- 역할 분리가 적절한가
- 입력과 출력이 자연스럽게 연결되는가
- 위임 가능한 판단 단위로 잘 설계되었는가

---

## 1. 테스트의 목적

- 스킬이 실제로 트리거되는가
- 스킬이 없는 경우보다 더 나은 결과를 만드는가
- 다양한 입력에서도 안정적으로 동작하는가
- 역할 위임 구조가 과하거나 부족하지 않은가
- 생성된 하네스가 사람이 검토 가능한 구조로 남는가

---

## 2. 테스트 방식

### 1) With-skill vs Without-skill 비교

같은 요청을 두 가지 방식으로 수행한다.

- With-skill: 스킬을 사용하는 경우
- Without-skill: 스킬 없이 수행하는 경우

### 비교 기준

- 결과의 정확성
- 구조의 일관성
- 설명의 명확성
- 재사용 가능성
- 산출물의 검토 가능성

### 목표

스킬이 없는 경우보다  
👉 **명확히 더 구조적이고, 더 재사용 가능하며, 더 검토 가능한 결과**를 만드는 것

---

## 3. 트리거 테스트

스킬은 description 기반으로 트리거된다.

따라서 다음을 반드시 테스트한다.

### 1) Should-trigger

스킬이 반드시 실행되어야 하는 요청

예:
- "이 저장소 구조 분석해줘"
- "프로젝트 구조 파악해줘"
- "코드베이스 이해하고 싶어"
- "하네스 역할 설계해줘"
- "로컬 스킬 구조 만들어줘"

### 2) Should-NOT-trigger

스킬이 실행되면 안 되는 요청

예:
- "피보나치 함수 만들어줘"
- "간단한 코드 한 줄 수정"
- "단일 파일 버그만 고쳐줘"

### 핵심

경계가 애매한 요청을 포함해야 한다.  
명백히 무관한 요청만 넣으면 테스트 가치가 낮다.

---

## 4. 드라이런 테스트

전체 하네스 흐름을 실제 실행 전 단계에서 검토한다.

### 확인 항목

- 단계 순서가 논리적인가
- 입력/출력 연결이 맞는가
- 중간에 끊기는 흐름이 없는가
- 역할 간 책임이 겹치지 않는가
- 결과가 파일 기반 산출물로 남는가

---

## 5. 역할 위임 구조 테스트

서브에이전트 활용을 염두에 둘 경우,  
역할 분리 자체도 테스트해야 한다.

### 확인 질문

- 이 역할은 한 문장으로 목적을 설명할 수 있는가
- 입력이 명확한가
- 출력이 명확한가
- 다른 역할과 지나치게 겹치지 않는가
- 굳이 분리할 필요가 없는 역할을 과하게 나누지 않았는가

### 좋은 분리 예

- domain-analyst: 저장소 분석
- harness-architect: 구조 설계
- qa-designer: 품질 질문 설계
- validator: 최소 구조 점검

### 나쁜 분리 예

- 저장소 분석 A
- 저장소 분석 B
- 저장소 분석 C

이처럼 구분 이유가 약하면  
역할 분리는 오히려 품질을 낮춘다.

---

## 6. 입력/출력 연결 테스트

좋은 하네스는 역할 간 연결이 자연스럽다.

### 확인할 것

- domain-analysis.md가 architecture의 실제 입력으로 기능하는가
- architecture.md가 scaffolding의 실제 기준이 되는가
- qa-strategy.md가 흐름과 검증 관점에 연결되는가
- orchestration-plan.md가 실제 협업 순서를 설명하는가

즉 단순히 파일이 존재하는지가 아니라  
👉 **실제로 다음 단계의 기준으로 기능하는지**를 봐야 한다.

---

## 7. 반복 개선 루프

테스트 후 반드시 개선을 반복한다.

### 과정

1. 테스트 실행
2. 문제 발견
3. 문제를 일반화
4. 스킬 수정
5. 재테스트

### 주의

- 특정 한 예시에만 맞춘 수정 ❌
- 여러 프로젝트에서도 통할 일반화된 개선 ✔

---

## 8. 자주 발생하는 문제

### 문제 1: 스킬이 트리거되지 않음
원인:
- description이 약함

해결:
- 더 구체적으로 작성
- 언제 써야 하는지 분명히 적기

### 문제 2: 결과 품질이 낮음
원인:
- 본문이 모호함
- 입력/출력 정의 부족

해결:
- 판단 기준 추가
- 입력/출력 섹션 명확화

### 문제 3: 특정 상황에서만 동작
원인:
- 과도한 특화

해결:
- 일반화

### 문제 4: 역할이 너무 많아 흐름이 복잡해짐
원인:
- 분리 기준이 약함

해결:
- 역할 통합 고려
- 파이프라인 단순화

### 문제 5: 파일은 있는데 실제 연결이 약함
원인:
- 산출물이 존재만 하고 흐름 기준이 되지 못함

해결:
- 역할 간 연결 설명 강화
- orchestration 문서 보강

---

## 9. 좋은 테스트의 조건

- 현실적인 요청 사용
- 다양한 표현 포함
- 경계 케이스 포함
- 역할 분리와 흐름도 함께 점검
- 파일 존재가 아니라 구조적 유효성을 본다

---

## 10. 범용 하네스에서 특히 중요한 점

범용 하네스 생성기에서는  
스킬 하나의 성능만 볼 것이 아니라,  
👉 **여러 역할이 함께 만들 구조의 품질**을 봐야 한다.

즉 테스트는 다음을 함께 본다.

- 개별 스킬 품질
- 트리거 적절성
- 역할 분리 적절성
- 입력/출력 연결성
- 전체 구조의 검토 가능성

---

## 11. 핵심 요약

- 스킬은 반드시 테스트해야 한다
- with/without 비교가 핵심이다
- 트리거 테스트가 중요하다
- 역할 위임 구조도 테스트 대상이다
- 입력/출력 연결성을 함께 봐야 한다
- 반복 개선이 필수다
