# Target Evaluation Playbook

이 문서는 실제 타겟 프로젝트에서 하네스를 재생성하고, 생성 결과가 운영 가능한지 평가하는 절차를 정의한다.

핵심은 단순히 파일이 생겼는지 보는 것이 아니라,
타겟 저장소 안에서 생성된 하네스가 실제로 `운영 가능 / 재작성 필요 / 재구성 필요` 중 어디에 가까운지 판정하는 것이다.

---

## 1. 언제 사용하는가

- 전역 하네스 생성기를 크게 변경한 뒤 타겟 프로젝트에서 결과를 다시 확인할 때
- phase 0~7 규칙이 실제 생성 결과에 반영되는지 확인할 때
- 보고서 품질, 다음 역할 안내, 재진입 규칙이 실제 저장소 문맥에서 살아나는지 점검할 때

---

## 2. 기본 절차

### 1) 타겟 프로젝트 준비

- 기존 하네스 산출물이 있으면 현재 상태를 먼저 백업하거나 커밋한다.
- 이번 검증이 `재생성`인지 `부분 재진입`인지 먼저 정한다.
- 상위 `AGENTS.md` 충돌 가능성이 있으면 먼저 그 상태를 기록한다.

### 2) 하네스 재생성 또는 갱신

- 신규 재생성이면 `harness-init.sh`부터 시작한다.
- 기존 구조 점검이면 시작 진입 역할 또는 `harness-update.sh`로 필요한 phase부터 다시 들어간다.
- 생성 직후에는 init 산출물을 완성본으로 보지 않고, 역할 작성이 끝났는지 먼저 본다.

### 3) 역할 작성 확인

- `domain-analysis.md`, `qa-strategy.md`가 실제 저장소 입력 문서처럼 읽히는지 본다.
- `harness-architecture.md`, `orchestration-plan.md`, `team-structure.md`, `team-playbook.md`가 하네스 운영 문서처럼 읽히는지 본다.
- 시작 진입 역할과 운영 감사 역할이 현재 상태와 다음 재진입 phase를 실제로 설명할 수 있는지 본다.

### 4) 검증 실행

- `harness-verify.sh`를 실행해 구조와 최소 규칙을 확인한다.
- verify 통과 후에도 바로 합격 처리하지 않고, phase 7 관점에서 품질 비교와 성숙도 평가를 한 번 더 남긴다.

### 5) 판정

- 아래 세 상태 중 하나로 결론을 남긴다.
  - `운영 가능`
  - `재작성 필요`
  - `재구성 필요`

---

## 3. 타겟 프로젝트 평가 체크리스트

### A. 입력 문서

- `exploration-notes.md`가 약한 입력 메모로 읽히고, 최종 분석 문서 행세를 하지 않는가
- `domain-analysis.md`가 실제 저장소의 제품/도메인/경계/실패 비용을 설명하는가
- `qa-strategy.md`가 추상적 QA 원칙이 아니라 현재 저장소의 검증 비용과 승격 기준을 설명하는가

### B. 메타시스템 문서

- `harness-architecture.md`가 역할 소개보다 패턴 선택과 다음 역할 기준을 먼저 설명하는가
- `orchestration-plan.md`가 요청 유형별 시작점과 재진입 기준을 설명하는가
- `team-structure.md`, `team-playbook.md`가 저장소 문서가 아니라 운영 문서로 읽히는가

### C. 에이전트 팀 규칙

- `AGENTS.md`, `.codex/config.toml`, `.codex/agents/*.toml`, `.codex/skills/*`가 같은 팀 규칙을 말하는가
- 프로젝트 맞춤 역할명이 실제 저장소의 요청 유형과 실패 경계를 설명하는가
- 시작 진입 역할이 현재 시작 역할, 다음 역할, 다음 재진입 phase를 실제로 제시할 수 있는가

### D. 검증과 재진입

- 운영 감사 역할이 단순 파일 검사자가 아니라 운영 기준 감사자로 읽히는가
- `운영 가능 / 재작성 필요 / 재구성 필요` 판단이 다음 action과 바로 연결되는가
- `재작성 필요`일 때는 `Phase 1~5` 중 어디로 가야 하는지 드러나는가
- `재구성 필요`일 때는 `Phase 0` 감사와 명시적 재구성이 먼저 제안되는가

---

## 4. 판정 기준

### 운영 가능

- verify를 통과한다.
- 입력 문서와 메타시스템 문서가 서로 다른 목적을 유지한다.
- 시작 진입 역할과 운영 감사 역할이 다음 행동을 모호하지 않게 설명한다.
- 보고서가 더 이상 범용 골격으로 읽히지 않는다.

### 재작성 필요

- 구조는 대체로 맞지만, 일부 문서가 가이드 문체나 일반론으로 흐른다.
- 시작점, 다음 역할 안내, 재진입 phase 중 하나 이상이 약하다.
- 운영 감사 역할이 특정 문서와 담당 역할을 다시 지정할 수 있다.

### 재구성 필요

- 문서 목적이 크게 뒤섞였거나, 역할 팀 규칙과 산출물이 서로 다른 구조를 말한다.
- update로 봉합해도 같은 문제가 반복될 가능성이 높다.
- phase 0 감사부터 다시 시작하는 편이 더 싸다.

## 5. 평가 결과 기록 형식

결과는 길게 쓰기보다 아래 순서로 남긴다.

1. 현재 판정: `운영 가능 / 재작성 필요 / 재구성 필요`
2. 가장 약한 축 1~2개
3. 다음 재진입 phase
4. 다시 써야 할 문서와 담당 역할
5. 비교 메모 한 줄

예:

- 현재 판정: `재작성 필요`
- 가장 약한 축: 입력 문서 품질, 다음 역할 명확성
- 다음 재진입 phase: `Phase 1`, `Phase 5`
- 다시 써야 할 문서와 담당 역할: `domain-analysis.md / 저장소 입력 문서 담당 역할`, `orchestration-plan.md / 흐름 조율 역할`
- 비교 메모: with-skill에서는 시작 역할은 선명하지만, 타겟 저장소 고유 흐름 설명은 아직 일반론으로 흐른다.

---

## 6. 환류 규칙

평가 결과는 진단으로 끝내지 않고, 반드시 어느 phase와 어느 규칙을 다시 고쳐야 하는지로 연결한다.

### A. 입력 문서 품질 문제

예시:

- `domain-analysis.md`가 일반론으로 흐른다
- `qa-strategy.md`가 저장소 고유 검증 비용을 설명하지 못한다

우선 재진입:

- `Phase 1`
- 필요 시 `Phase 5`

주 대상:

- 저장소 입력 문서 담당 역할
- 품질 전략 담당 역할

고칠 규칙:

- 입력 문서가 경로 나열보다 사용자 흐름, 경계, 실패 비용을 먼저 설명하는가
- 입력 문서와 메타시스템 문서의 목적이 섞이지 않는가

### B. 역할 설계 문제

예시:

- 역할명이 저장소 고유 용어보다 추상 역할명으로 흐른다
- 역할 수가 과하거나 부족해 다음 역할 연결이 불명확하다

우선 재진입:

- `Phase 2`

주 대상:

- 팀 구조 설계 담당 역할
- 흐름 조율 역할

고칠 규칙:

- `team-spec` 역할명 설계 근거
- 역할 경계, 중심/보조 역할, 요청 유형별 시작 역할 설계

### C. 생성 스킬 구체성 문제

예시:

- `.codex/skills/*/SKILL.md`가 이름만 다르고 본문은 비슷하다
- 시작 경로, 출력 템플릿, 재진입 규칙이 약하다

우선 재진입:

- `Phase 2`
- 이후 `Phase 3`

주 대상:

- `team-spec` 역할 카드
- 에이전트 정의 생성 단계

고칠 규칙:

- 대표 시작 경로
- 요청 유형별 하위 분기
- 산출 형식 템플릿
- 재진입 트리거
- 종료 판정 기준

### D. 운영 기준 문제

예시:

- `team-playbook.md`, `orchestration-plan.md`, `latest-session-summary.md`가 다음 재진입을 분명히 남기지 못한다
- `latest-session-summary.md`가 다음 시작 전에 다시 읽을 입력 파일과 최근 출력 파일을 남기지 못한다
- 운영 감사와 verify의 경계가 흐려진다

우선 재진입:

- `Phase 5`
- 필요 시 `Phase 6`

주 대상:

- 흐름 조율 역할
- 운영 감사 역할

고칠 규칙:

- 세션 종료 요약
- 다음 시작 역할 / 다음 재진입 phase
- 다음 시작 전 우선 확인 입력 파일 / 최근 출력 파일
- 운영 감사와 구조 검증의 책임 분리

### E. 재구성 필요 신호

예시:

- 입력 문서, team-spec, 생성 자산, 운영 문서가 서로 다른 팀 구조를 말한다
- update보다 새 구조 설계가 더 싸다

우선 재진입:

- `Phase 0`
- 이후 `Phase 2`

주 대상:

- 시작 진입 역할
- 팀 구조 설계 담당 역할

고칠 규칙:

- 상태 모드와 실행 패턴 재선정
- 역할 팀 전면 재설계
